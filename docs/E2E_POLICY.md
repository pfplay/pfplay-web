# E2E 테스트 정책

## 배경

PFPlay는 실시간 파티룸 음악 플랫폼으로, 다음 두 가지 구조적 특성이 있다.

- **REST + WebSocket 혼합**: 대부분의 행동은 REST API 호출이지만, 상태 변경 알림은 STOMP WebSocket으로 전달된다.
- **다중 사용자 실시간 동기화**: 두 클라이언트의 UI 상태가 WebSocket 이벤트를 통해 동기화되어야 한다.

이 문서는 **실제 서비스에서 실패하면 서비스가 망가지는 흐름**으로 기준으로 E2E 테스트 구축을 위해 기준과 방법, 그리고 최종 시나리오 목록을 정의한다.

## 규칙

**새 e2e 시나리오를 추가하기 전에 아래 체크리스트를 순서대로 통과해야 한다.**  
하나라도 탈락하면 e2e 추가를 중단하고 하위 레이어(단위/통합 테스트)로 위임한다.

## 선정 기준

### 기준 A: 실패 시 서비스가 완전히 멈추는 흐름

여러 시스템 경계를 순차적으로 통과하는 흐름으로, 단위 테스트로는 보장되지 않는다. 한 곳만 깨져도 사용자가 서비스를 아예 쓸 수 없게 됨

> PFPlay에서: Auth → 파티룸 입장 → WebSocket 구독 완료까지의 연결 체인

### 기준 B: 다중 사용자 간 상태 정합성이 핵심인 실시간 흐름

REST 응답과 WebSocket 이벤트가 순서와 타이밍을 지키며 조합되어야 하고, 두 클라이언트의 UI가 동기화되어야 한다. "두 브라우저가 실제로 같은 상태를 보는가"는 단위/통합 테스트로 검증할 수 없음.

> PFPlay에서: DJ 큐 등록 → 다른 사용자 화면에도 실시간 반영

### 기준 C: 사용자가 돌이킬 수 없는 상태 변이 흐름

한 번 실행되면 되돌리기 어렵고, 실패 시 데이터 불일치가 남는 흐름. 특히 종료·강퇴·페널티처럼 의도적인 파괴 행위가 포함된다.

> PFPlay에서: 파티룸 종료, HOST의 크루 강등/블록

### 기준 D: 외부 기기/서비스 의존 경계 흐름 — 주입으로 우회

Google OAuth, 블록체인 지갑(QR 스캔) 연결처럼 외부 기기나 서비스가 개입하는 흐름은 E2E에서 직접 실행할 수 없다. 대신 **주입/우회 후 그 이후부터** 검증한다.

| 외부 의존성                      | 우회 방법                                                |
| -------------------------------- | -------------------------------------------------------- |
| Google OAuth                     | dev-token 직접 발급, 인증 세션 수립된 상태에서 시작      |
| 블록체인 지갑 (RainbowKit/wagmi) | Playwright `addInitScript`로 `window.ethereum` mock 주입 |

## 선정 방법

기준 A~D를 통과한 후보에 아래 필터를 순서대로 적용한다.

```
Filter 1: 단위/통합 테스트로 보장 가능한가?
          → 가능하면 E2E 후보에서 제외. 하위 레이어에 위임.

Filter 2: 실패 시 영향 범위가 단일 사용자인가 vs 전체 파티룸인가?
          → 전체 파티룸에 영향 = 우선순위 UP

Filter 3: 발생 빈도 × 실패 비용
          → "자주 일어나고 실패하면 치명적" = 반드시 포함
          → "드물고 복구 가능" = 제외 우선순위 DOWN
```

필터 통과 후 **동일한 사용자 역할 여정으로 묶이는 시나리오는 반드시 통합**한다.  
총 E2E 시나리오는 최대 5개로 제한한다. 피라미드 상단이 무거워지면 CI 속도와 유지보수 비용이 급증한다.

## E2E 시나리오 선정 체크리스트

새 기능 추가 또는 기존 시나리오 재검토 시 아래 순서로 통과시킨다.

```
[ ] 1. 이 흐름이 실패하면 서비스가 죽거나 핵심 기능이 전체 불능이 되는가?
       → No이면 E2E 후보에서 제외. 하위 테스트에 위임.

[ ] 2. 단위 또는 MSW 통합 테스트로 충분히 보장 가능한가?
       → Yes이면 제외. E2E는 통합 테스트가 닿지 않는 곳만 커버한다.

[ ] 3. 실패 영향 범위가 단일 사용자인가 vs 파티룸 전체인가?
       → 전체에 영향이면 우선순위 UP.

[ ] 4. 발생 빈도 × 실패 비용이 높은가?
       → "자주 일어나고 실패하면 치명적"이면 포함.
       → "드물고 복구 가능"이면 제외 또는 Tier 2로.

[ ] 5. 외부 기기/서비스 의존성이 있는가?
       → Yes이면: 주입/우회가 가능한가?
           가능 → 주입 이후부터 검증하는 것으로 포함.
           불가 → 제외 후 수동 검증 목록에 추가.

[ ] 6. 이미 존재하는 E2E 시나리오 안에서 커버되는가?
       → Yes이면 별도 시나리오 추가 금지.

[ ] 7. 동일한 사용자 역할 여정으로 묶이는 시나리오가 있는가?
       → Yes이면 반드시 통합. 시나리오를 원자 단위로 쪼개지 않는다.

[ ] 8. 통과 후 총 E2E 시나리오 수가 5개를 초과하는가?
       → Yes이면 기존 시나리오 중 하위 테스트로 내릴 수 있는 것을 제거.
```
